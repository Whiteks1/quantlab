# QuantLab Rules (workspace)

- Keep architecture layered: data/, indicators/, strategies/, backtest/, execution/ (paper broker), reporting/.
- main.py is CLI entrypoint only; no business logic inside main.
- PEP8 + type hints + docstrings for public functions.
- No side effects on import; functions must be deterministic given inputs (seed randomness when used).
- Add/extend unit tests (pytest) for any new metric/logic; prefer small pure functions.
- Any change must include: (1) updated metrics output, (2) updated docs/README when behavior changes.
- Never remove existing CLI flags; deprecate instead.

# QuantLab Workspace Rules

Purpose:
Ensure safe, modular, reproducible development of the QuantLab trading research environment.

General Principles
- Keep code modular under `src/quantlab/`.
- Avoid placing logic in `main.py`; it should only orchestrate the pipeline.
- Prefer pure functions for analytics modules.
- Every new analytical module should be testable with pytest.

Data and Outputs
- Artifacts generated by the system must be written to the `outputs/` directory.
- Important artifacts include:
  - `outputs/trades.csv`
  - `outputs/trades_enriched.csv`
  - `outputs/report.md`
  - `outputs/report.json`
  - `outputs/equity.png`

Trading Safety
- The system must operate in **paper mode only** unless explicitly enabled otherwise.
- Do not implement live trading execution without explicit instruction.
- Never expose or commit API keys or secrets.

Testing
- All analytics logic should have unit tests using `pytest`.
- Tests should cover:
  - empty datasets
  - odd number of trades
  - repeated BUY/SELL sequences
  - deterministic outputs

Coding Style
- Use type hints where possible.
- Prefer pandas/numpy vectorization over loops when feasible.
- Avoid chained assignment in pandas.

Documentation
- Any analytics module must include docstrings explaining:
  - input format
  - output format
  - assumptions

Artifacts
- Every workflow must produce verifiable artifacts in `outputs/`.
