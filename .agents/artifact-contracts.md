# Artifact Contracts — QuantLab

## Purpose

This document defines the expected artifacts produced by QuantLab workflows.

Artifacts are the **source of truth** for experiment results and must remain stable across versions.

Artifacts must clearly communicate:

- what was executed
- which configuration was used
- what results were produced
- when the run occurred

Artifacts should be deterministic and reproducible.

---

# Run-Scoped Artifact Rule

All experiment artifacts should belong to a specific **run directory**.

Run directories live under:

outputs/runs/<run_id>/

A valid run directory must contain enough artifacts to reproduce the experiment context and results.

Plain `run` executions follow the same rule as `sweep`: successful executions must create
their own canonical directory under `outputs/runs/<run_id>/`.

Artifacts written outside a run directory should be considered **temporary or legacy** unless explicitly documented.

---

# Run Directory Structure

A typical run directory should follow this structure:

outputs/
└─ runs/
   └─ <run_id>/
      ├ metadata.json
      ├ config.json
      ├ metrics.json
      ├ report.json
      ├ run_report.md
      ├ config_resolved.yaml
      ├ leaderboard.csv / experiments.csv / walkforward.csv
      ├ trades.csv (optional)
      └ artifacts/ (optional)

Canonical run artifacts are:

- `metadata.json`
- `config.json`
- `metrics.json`
- `report.json`

Legacy artifacts such as `meta.json` and `run_report.json` are read-compatible only and should not be written by new run-producing flows.

The shared run registry is also part of the artifact surface for downstream tooling.
After every successful `run`, `sweep`, or `forward`, QuantLab refreshes:

- `outputs/runs/runs_index.csv`
- `outputs/runs/runs_index.json`
- `outputs/runs/runs_index.md`

The `artifacts/` folder can contain optional files such as:

- charts
- enriched datasets
- intermediate analytics

# Run Artifact Contracts

## outputs/runs/<run_id>/metadata.json

### Purpose

Stores identity and contextual metadata for a run.

### Generated by

Run system / execution layer.

### Minimum fields (implemented)

run_id  
created_at  
mode  
status  
command  
config_path  
config_hash  

### Optional fields

git_commit  
python_executable  
python_version  
request_id  
summary { ... }  
n_runs / n_train_runs / n_selected / n_test_runs  

---

## outputs/runs/<run_id>/config.json

### Purpose

Stores the exact configuration used for the run.

This ensures experiments are reproducible.

### Generated by

Run system / execution layer.

### Minimum fields (implemented)

ticker  
start  
end  
interval  
strategy parameters resolved for the run  

### Optional fields

fees_bps  
slippage_bps  
timeframe  
allocation_mode  
selection_rules  

---

## outputs/runs/<run_id>/metrics.json

### Purpose

Machine-readable summary metrics of a run.

Used for:

- run comparison
- ranking strategies
- automated analysis

### Generated by

Reporting / analytics layer.

### Minimum fields (implemented)

status  
summary { total_return, sharpe_simple, max_drawdown, trades, win_rate }  
best_result  
leaderboard_size  

### Optional fields

profit_factor  
expectancy  
volatility  
avg_holding_period  
exposure  

---

## outputs/runs/<run_id>/run_report.md

### Purpose

Human-readable report describing the run.

Used for:

- manual analysis
- research documentation
- experiment review

### Typical sections

Run configuration  
Performance summary  
Trade statistics  
Risk metrics  
Charts (optional)

---

## outputs/runs/<run_id>/report.json

### Purpose

Machine-readable report used for automation and research tooling.

### Minimum fields (implemented)

schema_version  
artifact_type  
status  
header { run_id, mode, created_at, git_commit, config_path, config_hash }  
artifacts [ ... ]  
summary { ... } or kpi_summary { ... }  

### Sweep machine-facing block

For run directories produced by `sweep`, `report.json` must expose:

machine_contract {
  schema_version,
  contract_type = "quantlab.sweep.result",
  command = "sweep",
  status,
  request_id?,
  run_id,
  mode,
  summary { total_return, sharpe_simple, max_drawdown, trades, win_rate },
  best_result?,
  artifacts { metadata, config, metrics, report }
}

For plain `run`, `report.json` is still canonical but does not currently include a dedicated
`machine_contract` block; integrations should use the canonical artifact pack plus session context.

# Legacy Artifact Contracts

Earlier workflows may still write artifacts directly under:

outputs/

Examples:

outputs/report.json  
outputs/report.md  
outputs/trades.csv  

These are considered **legacy outputs**.

The preferred structure is **run-scoped artifacts under outputs/runs/** using canonical artifact names.

Legacy read compatibility currently includes:

- `meta.json` -> canonical replacement: `metadata.json`
- `run_report.json` -> canonical replacement: `report.json`

---

# Artifact Stability Rule

Artifact schemas must not change silently.

Changes to:

- field names
- required columns
- artifact paths
- schema meaning

must be accompanied by:

1. update to this document  
2. entry in `decision-log.md`  
3. session log note if the change affects active work  

Backward compatibility should be preserved whenever practical.

---

# Design Philosophy

Artifacts are treated as **first-class outputs** of the research process.

QuantLab prioritizes:

- reproducibility
- traceability
- experiment comparability
- deterministic reporting

Artifacts should allow any run to be understood and reproduced without relying on chat history or memory.
